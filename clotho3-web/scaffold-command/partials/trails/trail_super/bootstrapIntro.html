Bootstrapping vs. Mixins

Bootstrapping an app is different from mixing in necessary components.

We have already see Mixins in several forms.

Bootstrapping does not rely necessarily on mixing in components - components may be added to the primary app stack (which is encouraged for functionality that can be shared), but here the widget is its own angular app.

This approach calls Application.script(), a cache-busting function to execute a script every time by time-stamping it. Continue to see examples.

Advantages

More Native
Adding components is not natively supported (though doesn't seem to cause problems)
Avoid Recompiling
Recompiling existing content will basically ruin the app. This way we know what we're inserting, rather than telling angular to compile a div that may have already been compiled.
Avoid Primary App $scope Lifecycle
Separating your widget from the parent app would avoid unnecessary processing / dirty-checking -- e.g. click listeners, timers, etc. set by other components.
Autonomy
App remains separate from the Primary app in several respects
Separate Functionality
If desired, components can be kept out of the Primary angular app.
Avoid Namespace Collisions
Namespace collisions are more likely to be avoided because they are not shared across apps.
Anti-paternalism
The route is more complicated, and probably will be less common, but why restrict it?

Disadvantages

Manual $scope destruction
Need to manually unwrap and destroy the app if it's not tied to a parent scope within container in which it is relevant.
Encourage Functionality in Main Stack
Bootstrapping apps separately may discourage developers from adding components directly to the parent stack, and keep them separate in their own module (don't think this will be a big deal though)
Services shared anyway
Services like PubSub, Collector, Socket etc. exist as per-page singletons, so there is little benefit in bootstrapping a separate widget in that respect. Currently, however, the ClothoAPI and ClientAPIs do not exist as singletons...
